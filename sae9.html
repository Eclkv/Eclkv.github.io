SAE 3.02 – Développer des applications communicantes

Conception et implémentation d'une architecture distribuée de routage en oignon : un système de communication anonyme entre clients via des routeurs virtuels, inspiré du protocole Tor.

Python Sockets TCP Threads Chiffrement asymétrique MariaDB PyQt6

Contexte et Objectifs

Cette SAE s'inscrit dans la ressource R3.09 (Programmation évènementielle) et vise à développer une application réseau complète. Le projet consiste à concevoir un système de communication anonyme entre plusieurs clients, à l'aide de routeurs virtuels interconnectés.

🎯 Mission principale

Implémenter un réseau distribué de routeurs virtuels permettant à des clients d'échanger des messages de façon totalement anonyme, grâce au routage en oignon : chaque routeur ne connaît que son voisin direct, et les données sont chiffrées par couches successives, comme dans Tor (The Onion Router).

Architecture du Système

L'architecture repose sur trois composants principaux qui communiquent via sockets TCP :

🖥️ Serveur Master

Superviseur central du réseau. Il reçoit les clés publiques de chaque routeur, définit la topologie et les distribue aux clients. Interface graphique PyQt6 pour visualiser les connexions et les logs en temps réel.

🔀 Routeurs Virtuels

Chaque routeur génère sa paire de clés publique/privée, s'enregistre auprès du Master, puis écoute les connexions entrantes. À la réception d'un message, il déchiffre une seule couche et transmet le reste au routeur suivant.

👤 Clients

Le client demande au Master la liste des routeurs disponibles et leurs clés publiques. Il choisit un chemin aléatoire (ex. 3 routeurs) et construit le message "en oignon" en chiffrant par couches successives avant d'envoyer le tout au premier routeur.

🗄️ Base de Données

MariaDB stocke les tables de routage, les clés publiques des routeurs et les logs anonymisés des connexions. Aucun routeur ne peut retracer l'origine complète d'un message.

Principe du Routage en Oignon

Le cœur du projet est le chiffrement par couches. Avant d'envoyer un message, le client construit un "oignon" de la façon suivante :

Construction du message chiffré (Client A → Client B via 3 routeurs)

Couche 1 : Enc(K1) → Adresse R2 + Couche 2
↓
Couche 2 : Enc(K2) → Adresse R3 + Couche 3
↓
Couche 3 : Enc(K3) → Adresse Client B + Message

Chaque routeur déchiffre uniquement sa couche avec sa clé privée, découvre l'adresse du prochain saut, et transmet le reste sans pouvoir lire le contenu. Résultat : aucun routeur ne connaît à la fois l'expéditeur et le destinataire.

Aspects Techniques Implémentés

🔌 Sockets TCP/UDP — Communication entre clients, routeurs et master (ports dynamiques, gestion multi-threads)
🧵 Threads — Serveur multi-clients non bloquant : chaque connexion entrante est traitée dans un thread dédié
🔑 Chiffrement asymétrique — Implémentation manuelle (sans librairie crypto) : paire clé publique/privée par routeur
📡 Routage — Table des prochains sauts stockée en BDD, appliquée selon les règles du Master
🗄️ MariaDB — Stockage des clés, routes et logs (requêtes SQL, connexion Python native)
🖼️ PyQt6 — Interface graphique pour le Master (visualisation réseau, logs) et le Client (envoi de messages)
📝 Journalisation anonymisée — Logs sans traçabilité complète de l'origine des messages

Difficultés Rencontrées

🔧 Défis techniques

L'implémentation manuelle du chiffrement asymétrique sans librairie dédiée est le défi principal : il faut concevoir un algorithme de chiffrement par couches fonctionnel et suffisamment robuste. La gestion des threads et des sockets simultanés (plusieurs routeurs actifs en parallèle) demande également une architecture soignée pour éviter les blocages et les conditions de course.

🏗️ Défi architectural

Coordonner le Master, les routeurs et les clients sur plusieurs machines virtuelles, tout en maintenant la cohérence des tables de routage en base de données et en garantissant que l'anonymat est préservé à chaque saut du réseau.

Project Summary (English)

🌍 SAE 3.02 – Communicating Application Development

Duration: ~65 hours | Format: Individual | Semester: 3rd

This project involved designing and implementing a distributed onion-routing architecture: an anonymous communication system between clients using virtual routers, inspired by the Tor protocol.

Technical Stack: Python (sockets, threading), manual asymmetric encryption (no crypto library), MariaDB, PyQt6 GUI, multi-VM deployment (Linux + Windows)

Key Concepts: Each message traverses a chain of virtual routers where every node only knows its direct neighbors. Data is encrypted in successive layers — each router decrypts only one layer, discovers the next hop address, and forwards the rest, ensuring no single node knows both sender and recipient.

Outcome: A fully functional distributed anonymous messaging system with a graphical master server for network visualization, multi-threaded virtual routers, and a PyQt6 client interface — deployed across multiple virtual machines to simulate a real distributed environment.
