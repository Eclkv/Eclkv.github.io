SAE 3.03 – Concevoir un Réseau Informatique Multi-Sites

Intégrateur réseau pour deux entreprises e-commerce (UC Exchange et ABC Conseil) présentes à Strasbourg, Nancy et Metz : architecture LAN hiérarchique, services réseau avancés (DNS, Web, Mail, AD) et interconnexion MPLS-VPN inter-AS via un FAI simulé sur EVE-NG.

VLAN / VRRP MPLS-VPN BGP / MP-BGP NAT/PAT DNS / Apache EVE-NG Windows Server Linux

Contexte et Objectifs

En tant qu'intégrateur réseau, nous avons été retenus pour concevoir et déployer l'architecture complète de deux entreprises e-commerce — UC Exchange et ABC Conseil — présentes chacune sur trois sites dans l'Est de la France : Strasbourg, Nancy et Metz.

🎯 Mission globale

Chaque site héberge 4 services (informatique, direction, financier, salle des serveurs). Le siège social de chaque entreprise dispose de serveurs DNS, DHCP, Web, Mail et Active Directory. Les sites sont interconnectés via une offre MPLS-VPN auprès d'un FAI composé de 3 Systèmes Autonomes (AS 10, 20, 30), simulé sur EVE-NG.

Chaque site abonne ses routeurs CE (Customer Edge) à deux FAI pour garantir la haute disponibilité. L'architecture LAN interne est hiérarchique (modèle three-tier / two-tier selon la taille du site), avec des commutateurs multicouches assurant le routage inter-VLAN.

Le Projet en Trois Parties

🔀 Partie 1 — Administration des LAN

Architecture commutée hiérarchique + Haute disponibilité

Mise en place d'une architecture commutée hiérarchique sur chaque site, avec des liens redondants entre commutateurs et le protocole Multi Spanning Tree (MST) pour éviter les boucles tout en conservant la redondance.

Création des VLANs par service (informatique, direction, financier, serveurs) en respectant le plan d'adressage global. Routage inter-VLAN opérationnel. Communication vers l'extérieur NATée via PAT. Haute disponibilité assurée par le protocole VRRP (Virtual Router Redundancy Protocol) en mode Actif/Passif.

🖥️ Partie 2 — Services Réseau

DNS + Web Apache + Mail SMTP/IMAP + Active Directory

Service DNS : Serveur primaire ns.ucexchange.com sous Windows Server 2019, serveur secondaire ns2.ucexchange.com sous Linux (ligne de commandes uniquement). Transferts de zone chiffrés avec DNSSEC. Accès SSH au secondaire filtré par sous-réseau ; les connexions refusées reçoivent un TCP reset pour tromper les scanners de ports.

Service Web : Apache hébergeant deux virtual hosts — un site intranet (accès LAN uniquement) et un site public (accès internet). Déployé sur le même serveur que le DNS secondaire.

Service Mail : SMTP + IMAP sur le serveur Linux. Comptes admin et client créés et testés via TELNET avec les commandes SMTP brutes.

Active Directory : Déploiement de l'AD sur Windows Server 2019 en utilisant le domaine DNS déjà établi. Ajout d'utilisateurs et connexion depuis un poste Windows client validée.

🌍 Partie 3 — Interconnexion MPLS-VPN

3 Systèmes Autonomes + BGP Route-Reflector + VRF

Maquette FAI complète sur EVE-NG avec 3 AS distincts :

AS 10 — Routage interne RIP, sessions BGP avec Route-Reflector (P1 = RR)
AS 20 — Routage interne RIP, sessions BGP avec Route-Reflector
AS 30 — Routage interne OSPF, sessions BGP avec Route-Reflector

Sessions MP-BGP configurées entre les ASBR (AS Border Routers) pour propager les routes VPNv4. VRF créées sur les PE (Provider Edge) pour isoler les deux clients :

ABC Conseil → VRF : RD 10:1 / RT 10:1 — routage PE↔CE via RIP
UC Exchange → VRF : RD 10:2 / RT 10:2 — routage PE↔CE statique

Les sessions BGP sont montées en peer-group pour simplifier la configuration. Seuls les préfixes clients transitent dans MP-BGP — les préfixes d'interconnexion FAI sont filtrés. Pour UC Exchange (parcours FI), translation NAT publique du serveur web de Metz exposé via l'adresse publique 11.11.11.1.

Configurations Clés

Haute disponibilité VRRP sur une interface VLAN :

interface vlan 10 ip address 10.242.11.252 255.255.255.0 vrrp 1 ip 10.242.11.254 ! IP virtuelle passerelle vrrp 1 priority 110 ! Routeur MASTER vrrp 1 timers advertise msec 100 vrrp authentication Finger

Configuration BGP avec Route-Reflector et peer-group (AS 10) :

router bgp 10 bgp router-id 10.10.1.1 ! neighbor IBGP peer-group neighbor IBGP remote-as 10 neighbor IBGP update-source Loopback0 neighbor IBGP route-reflector-client ! P1 = RR neighbor 10.10.2.2 peer-group IBGP neighbor 10.10.3.3 peer-group IBGP ! address-family vpnv4 neighbor IBGP activate neighbor IBGP send-community extended

Création de VRF pour le client UC Exchange sur un PE :

ip vrf UC_Exchange rd 10:2 route-target export 10:2 route-target import 10:2 ! interface e0/1 ip vrf forwarding UC_Exchange ip address 172.16.20.1 255.255.255.252 ! Lien vers CE

PAT vers internet sur le routeur de sortie :

ip nat inside source list 1 interface e0/0 overload access-list 1 permit 10.242.0.0 0.0.127.255 ! Tout le LAN UC Exchange ! interface e0/0 ip nat outside interface e0/1 ip nat inside

Tests de Validation

✅ LAN & VLAN

Vérification de la connectivité inter-VLAN, du routage via les switches de distribution, et du basculement VRRP en coupant l'interface du routeur MASTER.

✅ Services Réseau

Résolution DNS (nslookup), accès aux deux virtual hosts Apache, envoi de mail via TELNET SMTP, et connexion d'un poste au domaine Active Directory.

✅ Redondance MPLS

Coupure successive de PE2_ASBR2 puis ASBR4 : vérification que les pings inter-sites et vers le serveur web public restent opérationnels grâce à la redondance BGP.

✅ Filtrage BGP

Traceroute depuis UC Exchange Strasbourg → Metz : validation que le trafic passe bien par l'AS 20. Vérification que Metz↔Nancy pour ABC Conseil est bloqué (hub-and-spoke).

Compétences Développées

🔀 Architecture LAN avancée

Conception hiérarchique multi-sites, VLANs, trunks 802.1Q, MST, VRRP et routage inter-VLAN sur switches multicouches.

🌍 Réseaux opérateurs

Maîtrise de BGP, MP-BGP, MPLS-VPN, VRF, Route-Reflector et sessions inter-AS entre ASBR.

🖥️ Services réseau Linux/Windows

Déploiement DNS (BIND), serveur web Apache (virtual hosts), SMTP/IMAP (Postfix/Dovecot) et Active Directory sur Windows Server 2019.

🔐 Sécurité réseau

NAT/PAT, filtrage BGP par route-map, filtrage SSH par ACL avec TCP reset, DNSSEC et zone DMZ pour les serveurs exposés.

Bilan de l'Expérience

Cette SAE est sans doute la plus ambitieuse et la plus complète de ma formation. Elle m'a mis dans la peau d'un véritable intégrateur réseau face à un cahier des charges réaliste : deux clients, trois sites, un FAI multi-AS, des services métier complets et des contraintes de sécurité et de haute disponibilité à tous les niveaux.

La partie MPLS-VPN a été particulièrement enrichissante : comprendre comment BGP et MP-BGP permettent d'isoler les flux de deux clients sur la même infrastructure opérateur via les VRF est une compétence directement applicable en entreprise.

💡 Ce que j'ai retenu

Un réseau d'entreprise multi-sites, c'est bien plus que de la connectivité : c'est de la redondance à chaque couche, du filtrage précis des routes BGP, des services métier fiables et une DMZ bien dimensionnée. Chaque choix d'architecture a un impact direct sur la disponibilité et la sécurité de l'infrastructure.

Project Summary (English)

🌍 SAE 3.03 – Multi-Site Network Design

Team: Group of 3 | Platform: EVE-NG | Semester: 3rd

Acting as a network integrator, we designed and deployed the complete infrastructure for two e-commerce companies (UC Exchange and ABC Conseil) across three geographical sites (Strasbourg, Nancy, Metz).

LAN: Hierarchical switched architecture with VLANs, MST, inter-VLAN routing, PAT and VRRP high availability. Services: DNS (BIND + Windows Server 2019), Apache dual virtual host, SMTP/IMAP mail server and Active Directory. WAN: Full ISP simulation on EVE-NG with 3 AS (RIP/OSPF internally), MP-BGP with Route-Reflectors, MPLS-VPN with per-customer VRFs (RD/RT 10:1 and 10:2), BGP filtering for traffic engineering, and public NAT for web server exposure.

Key Outcome: End-to-end validated multi-site infrastructure with redundancy tested at every layer — from VRRP failover on LANs to BGP path redundancy across ISP AS boundaries.
